* Overview
This file tracks the forward plan for the st-suite, plus open questions and
cleanups discovered while documenting the existing code. See [[file:AGENTS.md][AGENTS.md]] for
the suite-level orientation.

* GUI [/]
A unified GUI is the next major piece of work. It will sit alongside the
existing CLI-launched JACK clients and provide visual control/feedback for the
whole suite.

** Decisions
- *Toolkit:* =eframe= (egui), following the conventions of the
  =qwertysynth= project. Use eframe's default glow backend unless a reason
  to switch to wgpu appears.
- *Reference codebase:* =./qwertysynth/src/gui/= (a sibling project, not
  part of st-suite — being removed after the GUI plan is captured here).
  Conventions extracted below in [[*Conventions derived from qwertysynth][Conventions derived from qwertysynth]].

** Conventions derived from qwertysynth
All adopted as the house style for st-suite GUI work unless explicitly
overridden below.

*** Crate choice & versions
- =eframe= (not raw egui). *Pin =eframe = "0.34"= per-crate* (matches
  qwertysynth and verified by the st-loop spike). Bump deliberately;
  the trait surface has changed across recent versions (see below).
- Default backend (glow). No wgpu, no =egui_extras=, no =rfd= unless a
  concrete need appears.
- =crossbeam-channel= for all GUI ↔ worker thread messaging.

*** eframe 0.34 API specifics (learned from the st-loop spike)
- =impl eframe::App= must define
  =fn ui(&mut self, ui: &mut egui::Ui, frame: &mut eframe::Frame)=.
  =fn update(&mut self, ctx: &egui::Context, frame: &mut eframe::Frame)=
  exists but is =#[deprecated]= in 0.34 — do *not* use it.
- Inside =ui()=, panels are added with =.show_inside(ui, ...)=, not
  =.show(ctx, ...)=. Use =Panel::top("id")= / =Panel::bottom("id")= /
  =Panel::left("id")= / =Panel::right("id")=; the standalone
  =TopBottomPanel= / =SidePanel= type aliases are deprecated.
- The =run_native= app-builder closure must return
  =Result<Box<dyn App>, Box<dyn Error + Send + Sync>>=. Wrap in =Ok=:
  #+begin_src rust
  Box::new(move |cc| {
      cc.egui_ctx.set_visuals(egui::Visuals::dark());
      Ok(Box::new(MyApp::new(...)))
  })
  #+end_src
- Import =egui= via =use eframe::egui;= (eframe re-exports it).
- Reference implementation: =st-loop/src/bin/gui_spike.rs= (on the
  =dev= branch). Delete or absorb once real GUI implementation begins.

*** Process / window model
- *Single native window per process,* launched via =eframe::run_native=
  on the main thread. Pop-out config UIs are =egui::Window= opened from
  the central area, *not* additional OS windows.
- =ViewportBuilder::default().with_inner_size([W, H]).with_title(...)=,
  dark visuals (=cc.egui_ctx.set_visuals(egui::Visuals::dark())=) set
  once at construction.
- =main()= is synchronous. If async tasks are needed (JACK, evdev, NSM,
  ...), construct a =tokio::runtime::Runtime= manually and =rt.spawn(...)=
  them before calling =eframe::run_native=.

*** File / module organization
- One module per panel / widget / indicator under =src/gui/=, named by
  suffix:
  - =<thing>_panel.rs=    — composite interactive UI, may own a popup
                            config window.
  - =<thing>_widget.rs=   — larger drawable widget (custom paint, takes
                            real estate).
  - =<thing>_indicator.rs= — tiny status dot / lamp.
  - =<thing>_bar.rs=      — full-width strip (info/status).
- =gui/mod.rs= contains *only* glue: submodule decls, the =App= struct,
  the =eframe::App= impl, the =run()= launcher, and per-frame
  drain/dispatch helpers. *No business logic in =mod.rs=.*

*** Panel / widget implementation pattern
- *No trait, no struct-with-show.* Each module exposes free functions:
  - Read-only:   =pub fn show(ui: &mut Ui, state: &AppState)=
  - Interactive: =pub fn show(ui: &mut Ui, state: &mut AppState) -> R=
    where =R= is whatever the caller needs (=bool= "needs rebuild",
    =Option<NewValue>= for the new param to forward on a channel, ...).
- Panels with a popup config window use the *two-function pattern*:
  #+begin_src rust
  pub fn show_top_bar(ui: &mut Ui, state: &mut AppState) -> bool;
  pub fn show_config_window(ctx: &Context, state: &mut AppState) -> bool;
  #+end_src
  The config window takes =&Context= (not =&mut Ui=) because
  =egui::Window= opens against the context.
- *All widget state lives in =AppState=* — including transient flags like
  =show_<x>_config: bool=. Widgets are pure renderers; nothing persists
  inside them. (Rationale: everything inspectable, trivially serializable
  later if we ever need to round-trip UI state via NSM.)
- Give each popup =egui::Window= a stable =Id= derived from the active
  variant (=Id::new("<panel>_window_<variant>")=) so egui doesn't lose
  layout when the user switches modes.

*** AppState pattern
- A single =struct AppState= holds *all* app state.
- =pub type SharedState = Arc<Mutex<AppState>>;=.
- *The audio / JACK thread never locks =AppState=.* It owns its own
  data and communicates exclusively via channels. The mutex exists for
  GUI-thread + helper-task contention only.
- For "mode" data with per-mode parameters, use an *enum-of-param-structs*:
  one enum variant per algorithm, each carrying its own typed param
  struct. Provide =name() -> &'static str= on the enum for labels.
  Match the variant in both the renderer and the worker.

*** GUI ↔ worker communication
- *=crossbeam_channel::unbounded= exclusively.* No =tokio= channels in
  the GUI <-> audio path.
- *Push, drained by =try_recv= once per frame.* At the top of
  =eframe::App::update=, call a =drain_<x>_events= helper that
  =try_recv=s every relevant channel into local state under a short
  lock, then releases the lock before drawing.
- *Decode/translate in the worker, send high-level state to the GUI* —
  never raw MIDI bytes or raw transport frames. e.g. the audio thread
  decodes sustain CC 64 into a =bool= before pushing.
- The audio / RT thread does only lock-free crossbeam sends; heavy work
  (e.g. recomputing a wavetable, rebuilding a scene) happens on the
  *GUI thread* and the result is pushed back as a finished value the
  audio thread atomically swaps in.
- =ctx.request_repaint()= at the end of every =update()= keeps the GUI
  live so it polls channels each frame. Simple but burns CPU; if a panel
  is idle-heavy, switch to =request_repaint_after(Duration)= or convert
  that surface to event-driven repaints.

*** Layout idioms
- Prefer =TopBottomPanel::bottom("...")= for status strips; draw the main
  content directly in the central area. Use =SidePanel= only when there's
  a real reason.
- Custom drawing: =ui.allocate_exact_size(...)= then
  =ui.painter().rect_filled / circle_filled / rect_stroke / text=.
- Pull colors from =ui.visuals()= (=widgets.inactive.bg_fill=,
  =selection.bg_fill=, =text_color=) so widgets respect the theme.
  Hardcoded RGBs allowed only for status indicators (red/green lamps).
- Fixed sizing constants at the top of each widget file, with helper
  fns for derived sizes.
- =ComboBox::from_id_salt("<name>_combo")= for stable IDs.
- Time-valued sliders: =.logarithmic(true).text("...").suffix(" s")=.

*** "Needs work" return convention
Panel/widget functions return =bool= or =Option<Value>= indicating
whether the caller should kick off heavy work. =App::update= aggregates
these flags and triggers the rebuild *once at the end of the frame*,
not inline during widget rendering.

*** House style
- Tab indentation (matches qwertysynth and the rest of st-suite).
- TODO marker style: inline =// TODO: <concrete next step>= comments
  with actionable descriptions, not vague aspirations.

** Architecture for st-suite

*** Scope decision (defer — see open questions)
Two viable shapes for the suite GUI. *Pick one before implementation*:
- *(A) One GUI process per component.* Each of st-conductor / st-click
  / st-loop grows its own =--gui= mode (or sibling binary) that opens
  an eframe window owned by that process. Mirrors qwertysynth (process
  owns both audio + GUI). Natural NSM fit: each component is one NSM
  client with an =:optional-gui:= capability.
- *(B) One unified GUI process.* A new =st-gui= submodule with one
  eframe window that talks to all three running components over some
  control bus. The components stay headless. Better single-pane-of-glass
  UX, but adds a control protocol design problem (see next).

Strong lean: *(A)* for the first cut. It matches qwertysynth, avoids
inventing a control bus, keeps audio + GUI in the same process (so the
GUI can use Arc<Mutex<AppState>> + crossbeam exactly as qwertysynth
does), and gives us three small wins instead of one big one. (B) can
be layered on top later as an additional aggregator window.

*** Per-component GUI plans (assuming scope A)

**** st-conductor GUI
- Transport panel: tempo (BPM), time signature (numerator/denominator),
  current BBT readout, transport state (Rolling / Stopped / Starting).
- Beat indicator: a flashing dot driven by the same =next_beat_frame=
  the conductor pushes to st-sync.
- Optional: a sparkline of frames-per-beat over time for debugging.
- Channels: the JACK timebase callback already updates an
  =Arc<AtomicU64>=; add an =Arc<AtomicXxx>= or crossbeam channel for
  current BBT / state and have the GUI poll once per frame.

**** st-click GUI
- Sequence selector: combo box listing top-level keys from the loaded
  YAML.
- Beat indicator: same flashing-dot pattern as the conductor, driven
  from =st_sync::client= ticks.
- Tempo / meter readout: read-only, mirrored from JACK transport via
  =st_lib::jack_transport::query_transport= once per frame.
- YAML editor (later): an =egui::TextEdit::multiline= bound to the
  current config file with a "Reload" button.

**** st-loop GUI
- 8×8 scene/track grid (the core view). Each cell shows record/play
  state and (later) a level meter.
- Track strip: input gain, output level, mute / solo.
- Scene controls: play / stop, undo, clear, go — duplicating the MIDI
  PC vocabulary so a touch screen / mouse can drive the looper.
- Transport readout: bar / beat from JACK transport.
- NSM-aware: respect the path NSM provides; "dirty" flag flips on
  state changes.

*** Control / sync paths (scope A specifics)
- *No new control protocol.* Each component's GUI talks to its own
  audio thread via crossbeam, exactly as qwertysynth does.
- *Cross-component awareness:* GUIs can read JACK transport (tempo /
  BBT) via =st_lib::jack_transport= and the next-beat tick via
  =st_sync::client::Client=. That covers most of what each GUI needs
  to mirror the others. No bespoke RPC needed for v1.

** Open questions (must resolve before implementation)
- [X] *Confirm scope A vs scope B* — *A* (per-component GUIs). The rest
  of this plan assumes A.
- [X] *Where does each GUI live?* — *A1*: new =src/gui/= module inside
  each existing submodule (=st-conductor=, =st-click=, =st-loop=).
  Same binary; eframe runs on main thread; audio task spawned.
- [X] *Does each GUI run by default?* — *NSM =:optional-gui:=*
  capability is the primary mechanism: NSM toggles the GUI on/off via
  the standard =/nsm/client/show_optional_gui= /
  =/nsm/client/hide_optional_gui= messages, and the component must
  reply with =/nsm/client/gui_is_shown= or =/nsm/client/gui_is_hidden=.
  Also expose an *optional =--no-gui= CLI flag* that disables the GUI
  entirely for headless / scripted runs (independent of NSM).
  - Default behavior when launched by NSM: GUI hidden until NSM asks.
  - Default behavior when launched manually (no =NSM_URL=): GUI shown
    unless =--no-gui= is passed.
- [X] *eframe version to pin.* *=eframe = "0.34"=* per the st-loop
  spike. Bump deliberately; the trait API has churned recently.
- [X] *Threading model for st-loop* — see the spike in
  =st-loop/src/bin/gui_spike.rs= (commit on =dev= branch). The
  qwertysynth pattern composes cleanly with st-loop's existing tokio
  use: build a multi-thread =tokio::runtime::Runtime= manually,
  =rt.spawn(...)= the async tasks (NSM, dispatcher, jackio), then
  call =eframe::run_native= on the main thread. Spike verified
  successful build with eframe 0.34 alongside tokio "full" and the
  existing st-loop deps.
- [ ] *Repaint cadence.* qwertysynth =request_repaint()= every frame
  unconditionally. For the looper grid + level meters this is fine; for
  st-conductor's mostly-static tempo display, =request_repaint_after=
  may be friendlier. Decide per-panel.
- [ ] *Beat indicator data source.* Two options for getting "beat
  happened now":
  - Each GUI runs its own =st_sync::client::Client= and pushes ticks
    into its app state via a channel (same as st-click and st-loop
    already do for their audio paths).
  - The existing audio-side =Client= forwards ticks to the GUI via a
    new crossbeam channel.
  - Recommendation: *the latter* — don't open a second TCP connection
    per process.
- [ ] *Should the GUI panels be tested?* qwertysynth has none. egui has
  =egui_kittest= for snapshot tests but it's a heavy dep. Decision:
  *no automated GUI tests for v1*; rely on the existing pure-logic
  test suite for the underlying state.

* NSM (Non Session Manager) [/]
Each st-suite component should be individually NSM-compatible so the suite
can be started/saved/restored as a single session by an NSM host
(=nsm-legacy-gui=, =agordejo=, =RaySession=, ...).

  =dispatcher.rs:process_save_request/process_load_request=). Audit:
  - [ ] Does Save actually persist *audio* (WAVs per recorded sequence)
    alongside =config.yaml=, or only metadata? The dispatcher writes
    sequence metadata (track, beats, filename); confirm WAV bytes are
    written and re-read correctly on a round-trip.
  - [ ] Verify NSM Announce reply / capabilities are correct
    (=:switch:= / =:dirty:= / =:optional-gui:= flags as appropriate).

  tempo, time signature (and any future transport state).
