* Overview
This file tracks the forward plan for the st-suite, plus open questions and
cleanups discovered while documenting the existing code. See [[file:AGENTS.md][AGENTS.md]] for
the suite-level orientation.

* Timing & sync architecture — beat-window protocol [/]
Foundational rework of the contract between st-conductor and st-sync
clients. Triggered by an OOM crash in st-click (sized a frame-indexed
buffer using =next_beat_frame= as if it were =frames_per_beat=, which
only coincide when transport starts near frame 0). The deeper problem
is that every client today re-derives timing facts (=frames_per_beat=,
fractional beat position, etc.) from JACK transport using slightly
different math, which drifts. Fix: make the conductor publish those
facts authoritatively over st-sync, and refactor consumers to be
beat-indexed rather than frame-indexed so tempo changes track
continuously without rebuilds.

** Background — what's broken today
- *st-click crash*: =Sequence::new(beats_per_bar, next_beat_frame, 1)=
  at =st-click/src/sequencer.rs:195= (and =:248= for runtime swaps)
  passes =next_beat_frame= (absolute JACK sample position) where
  =frames_per_beat= is expected. Allocates =beats_per_bar *
  next_beat_frame * bars / 2= slots. With the conductor running for
  even a minute before st-click connects, this is tens of millions of
  entries; runtime sequence swaps re-allocate at ever-larger sizes.
- *st-click ignores tempo changes entirely*: the buffer is sized once
  at startup and never rebuilt on tempo change. Even if the value
  above were correct, a tempo change mid-session would silently
  desync the click from the conductor.
- *st-click ignores meter changes entirely*: =beats_per_bar= is
  captured once at startup; per-cycle JACK transport queries discard
  =snap.beats_per_bar=.
- *st-loop is partially safe*: playback is beat-counted (loop wrap is
  driven by =observe_beat=, not by frame count), so tempo changes
  don't break loop boundaries. But =stop_recording= uses
  =self.beats_per_bar= captured at sequence construction
  (=st-loop/src/sequence.rs:105=) instead of the live value from
  =JackSyncFanoutMessage.beats_per_bar=, so meter changes mid-
  recording round to the wrong bar length.
- *Cross-client drift risk*: nothing forces every client to use the
  shared =st_lib::beat_math::frames_per_beat= helper. Whenever a
  client computes =frames_per_beat= locally from BPM + sample rate,
  rounding differences vs. the conductor's own computation can
  accumulate.

** Protocol — sliding window of beat frames
Replace the current single-=u64= st-sync wire format with a
*length-prefixed window* of absolute beat frames, ordered oldest
first:

#+begin_src
  message := u8 length, length * u64 frame
#+end_src

The window's first entry is the most recently completed beat; the
last is the furthest-future beat the conductor has committed to.

*** Contract — guarantees the conductor makes
1. *Immutability.* Once a beat frame appears in any published
   window, that frame is fixed forever. The conductor may not
   retroactively change the frame position of any published beat,
   regardless of tempo changes from any source (user, score, queue).
2. *Monotonicity.* Beat frames are strictly increasing within a
   window, and across windows. The transport never moves backward.
   st-suite is a *live performance environment*; there is no
   scrubbing, no jumping forward, no rewind. Musical repeats are
   possible but must inherently occur at future frames.
3. *Forward extension only.* Successive windows may add new
   beats at the high end and drop old beats at the low end (sliding
   forward), but never rewrite middle entries.

*** Contract — what clients may assume
- =frames_per_beat= for any pair of adjacent published beats is
  *exact* and *agreed-on by every client*: it's just the subtraction
  of two facts received over the wire.
- =beat_position_at(frame)= for any frame whose surrounding beats
  are in the window is well-defined and stable: linear interpolation
  between adjacent published beats.
- Beat positions inside the window will *not change*. Events
  scheduled against the window will not drift, double-fire, or be
  missed due to retroactive tempo updates.
- The audio thread may snapshot or recompute beat ↔ frame
  conversions freely without worrying about consistency.
- The transport will never move backward. Components do not need
  defensive code for rewind, scrub, or jump-back; if a frame from
  the past is needed, the component is responsible for caching it
  (e.g. st-loop already does this implicitly by recording into a
  buffer).

*** Window size
- Default: =max(4, beats_per_bar + 1)=. A floor of 4 ensures even
  very small meters get =prev + current + next + one extra=; a
  ceiling of =beats_per_bar + 1= caps the commitment horizon at
  roughly one bar so user tempo changes feel live.
- Fixed at compile time for v1; negotiable at handshake later if
  needed.

*** Tempo change policy
The conductor honors the immutability guarantee by deferring any
tempo change to take effect *after the last beat already on the
wire*. Concretely, when a new tempo is requested (user input, score
event, queue pop):
- *Policy 3 from design discussion*: honor the existing window;
  stop extending it until the change's effective beat is reached,
  then resume publication using the new tempo. Clients see the
  window naturally shrink and re-grow.
- *Never truncate or rewrite the published window* — that would
  violate immutability.

*** Conductor internal model
- Build a *score-shaped* tempo envelope: =Vec<(beat_index, bpm,
  beats_per_bar)>=. The conductor's beat-publisher consults the
  envelope when extending the window.
- Live user input (tap tempo, "set tempo to X") becomes an
  append-to-end of the envelope.
- Queue-style operation (queue a tempo change at beat N) is a
  degenerate one-point envelope addition.
- A future scored ritardando is a series of per-beat envelope
  points expressing the tempo curve.

The wire protocol does not distinguish these — they all reduce to
"sequence of beat frames." The score model is the implementation
choice; the protocol is agnostic.

*** Protocol versioning
- Single coordinated cutover for v1 (all crates in monorepo).
- Add a *protocol version byte at connect* the moment a non-monorepo
  consumer appears, to enable graceful handshake. Cheap insurance.

*** st-click [/]
*** beat-indexed Sequence refactor [/]
The current frame-indexed buffer is the root cause of the OOM and
makes continuous tempo tracking impossible. Replace with a
beat-coordinate event list scheduled per-cycle from the live
=frames_per_beat=.

- [ ] New representation:
  #+begin_src rust
  struct Sequence {
      beats_per_bar: f32,
      bars: u32,
      // Sorted by event_beat. (event_beat, midi_bytes)
      events: Vec<(f64, OwnedMidiBytes)>,
  }
  #+end_src
- [ ] =add_notes= and =add_notes_with_offset= produce
  =(event_beat, bytes)= entries; never reference frames.
- [ ] New per-cycle scheduler:
  - Track =last_cycle_end_beat: f64= and =next_event_index: usize=.
  - Per cycle: compute =cycle_end_beat = client.beat_position_at(
    cycle_end_frame)=; advance cursor through events in
    =(last_cycle_end_beat, cycle_end_beat]=; emit each with sub-
    cycle MIDI =time= computed from the same =frames_per_beat=.
  - Wrap cursor and =last_cycle_end_beat= when crossing
    =bars * beats_per_bar=. Loop may wrap multiple times in one
    cycle at very fast tempos.
- [ ] Tests covering the four invariants and six failure modes from
  the design discussion:
  - [ ] Float precision: event at integer beat fires exactly once
    across 10,000 varied-length cycles.
  - [ ] Sequence wrap: cycle spanning multi-bar loop boundary fires
    events on both sides.
  - [ ] Pathologically long cycle: 10 events worth of cycle fires
    all 10.
  - [ ] Continuous tempo change: stepwise =frames_per_beat=
    increments across cycles, assert per-beat hit count stays
    correct and wall-clock spacing increases monotonically.
  - [ ] Tempo change at cycle boundary: assert no event near the
    boundary is missed or doubled.
  - [ ] Late client connect: =Sequence::new= called when window's
    first entry is at frame 4,800,000 (i.e. conductor's been
    running). Assert buffer allocation is bounded by note count,
    not by frame count.

* st-click — schema, drumkv1 pairing, sequence selection [/]
Forward plan for the metronome / polyrhythm generator. The pure-logic side
(=Sequence::add_notes=, =get_beat_value=, =tuplet=) is already test-covered
(=st-click/src/sequencer.rs= tests). What follows extends the YAML schema,
nails down the drumkv1 patch this feeds into, and rationalizes how a user
picks a sequence (CLI / GUI / NSM).

** YAML schema evolution [/]
Two additive primitives plus one structural change. All three preserve
backward compatibility with the current shipped example.

*** Add =offset= (per-note, signed, in beats) [/]
- Field: =offset: f64= (optional, default =0.0=), measured in
  *quarter-note beats* — independent of the entry's =beat_value=, so it
  composes naturally with =tuplet=.
- Semantics: applied after =skip=; conceptually shifts every emitted index
  by =offset * frames_per_beat / 2= frames, wrapping modulo =length=.
- Negative values legal (anticipations / push-beats).
- Validation: reject =|offset| >= bars * beats_per_bar= at parse time.
- Tests:
  - quarter at =offset: 0.5= in 4/4 with =frames_per_beat=100= → frames
    =[25, 75, 125, 175]=.
  - negative wrap: =offset: -0.25= equals =offset: 3.75= for
    =bars: 1, beats_per_bar: 4=.
  - composes with =tuplet=: triplet quarter at =offset: 1.0= stays a
    triplet, just shifted.

*** Add =bars: N= per-sequence (Option A schema extension) [/]
Currently =Sequence::new= already accepts =bars: u32= and the wrap math
(=n_beats = beats_per_bar * bars=) is wired through, but the call site is
hardcoded to =1= (=st-click/src/sequencer.rs:157=) and there is no YAML
surface. Fix by promoting each sequence value from "list" to "mapping with
optional metadata":

#+begin_src yaml
# Old form still works (implicit bars: 1):
backbeat:
  - { note: "C-1", beat_value: 1.0, every: 2 }
  - { note: "D#/Eb-1", beat_value: 1.0, every: 2, skip: 1 }

# New form:
rumba_clave_4bar:
  bars: 4
  notes:
    - { note: "C#/Db-1", beat_value: 16.0, offset: 0.00 }
    - { note: "C#/Db-1", beat_value: 16.0, offset: 1.00 }
    ...
#+end_src

Parser detects list-vs-mapping at the sequence root. Touches =config.rs=
and the =Sequence::new(.., 1)= call site only.

*** Refresh shipped example [0/2]
- [ ] Add 6–8 curated sequences grouped by header: common / odd meter /
  polyrhythm / multi-bar. One-line musical-intent comment per sequence
  (not just the math).
- [ ] Add a header block at the top of the YAML pointing at the README and

  itself is what users edit.)

*** Schema validation pass [/]
- [ ] Reject unknown keys (typos like =beatvalue= silently no-op today).
- [ ] Reject =offset= that wraps past the sequence span.
- [ ] Reject negative =every= / =skip=.
- [ ] Surface YAML parse errors with file path + line, not =unwrap()= panic.

1. =offset= primitive (purely additive; old configs unchanged).
2. =bars: N= via Option-A schema extension.
3. Schema validation pass (depends on the new parser structure from 2).
4. Refresh shipped example.

** drumkv1 patch / kit pairing [/]
=st-click= emits MIDI on =st-click:midi_out=; the audible click is whatever
synth that's wired to. The intended pair is =drumkv1= (DPF / LV2 / JACK
drum sampler) with a small fixed kit so the YAML's note names map to
musically appropriate sounds.

*** Note-to-sample mapping (canonical kit)
The shipped example uses =C-1=, =D#/Eb-1= (one click + one accent). For
the expanded example library we need a slightly larger vocabulary. Propose
a fixed mapping in =etc/st-tools/= shipped alongside the YAML:

| MIDI note  | Role            | Suggested sample type           |
|------------+-----------------+---------------------------------|
| =C-1=      | low click       | woodblock low / kick            |
| =C#/Db-1=  | downbeat accent | bell / cowbell                  |
| =D-1=      | mid click       | woodblock mid / snare           |
| =D#/Eb-1=  | secondary click | rim / clave                     |
| =E-1=      | accent 2        | high bell                       |
| =F-1=      | sub-click       | hat closed                      |
| =F#/Gb-1=  | hat / shaker    | hat open / shaker               |
| =G-1=      | spare           | (user)                          |

Rationale: stays within MIDI octave -1 so it matches what already ships,
leaves the rest of the keyboard untouched for users who want to layer
=st-click= over a real kit.

*** Deliverable: drumkv1 preset
- [ ] Ship =etc/st-tools/st-click.drumkv1= (drumkv1's =.drumkv1=
  preset format — XML-ish, contains the sample paths and per-pad params).
- [ ] Source the actual sample files. Candidates:
  - Synthesize tiny WAVs (sine click + filtered noise) at build time;
    keeps the suite self-contained, no licensing concerns.
  - Pull from a CC0 sample pack and vendor under =etc/st-tools/samples/=.
  - Recommendation: *synthesize*. A 50-line Rust helper (=examples/=
    binary?) generating the eight WAVs is more honest than vendoring
    a sample pack, and the timbres can be tuned to be unambiguous as
    a metronome (short, dry, pitched apart).
- [ ] Document the wiring in =st-click/README.md=: launch drumkv1 with
  this preset, =jack_connect st-click:midi_out drumkv1:in=, done.

*** Open questions
- [ ] Should =st-click= optionally *launch* drumkv1 itself (subprocess)
  when not under NSM, or always assume the user / NSM did it? Lean:
  *always assume external*. Keeps st-click a single-purpose tool.
- [ ] Does the canonical kit go in =st-lib=, =st-click=, or its own
  =etc/st-tools= subdir? Lean: =st-click/etc/st-tools/= alongside the YAML.

** Sequence selection: CLI / GUI / NSM [/]
Today the sequence name is a required positional CLI arg (=st-click/src/main.rs:14=)
and the GUI combo box is read-only after launch (=st-click/src/gui/sequence_panel.rs:31=
warns "runtime sequence swap not yet implemented"). Fix all three together.

*** Make the CLI argument optional
- =sequence_name= becomes optional. Resolution order:
  1. Explicit CLI positional → use it.
  2. NSM session state (if =NSM_URL= set and a saved state exists) → use it.
  3. First sequence in the loaded YAML → use it (with a stderr note).
  4. If YAML is empty → hard error.
- =--no-gui= already exists; keep it.
- Add =--list= to print sequence names and exit (handy for shell completion
  and for sanity-checking the config file).

*** Runtime sequence swap (GUI-driven)
Currently impossible because =Sequencer::start= consumes the sequence
name once and loops forever. Refactor:

- =Sequencer= grows a =Receiver<SequencerCommand>= where
  =SequencerCommand::SwitchTo(String)= is the only variant for v1.
- On each iteration of the main loop, =try_recv= the command channel
  before the JACK transport query. On =SwitchTo(name)=:
  1. Rebuild a fresh =Sequence= from the config + new name.
  2. Snap the playhead to the next bar boundary (using the
     =beat_counter % beats_per_bar = 0= check already present at
     =st-click/src/sequencer.rs:188=) so the swap is musical, not jarring.
  3. Update =active_sequence= via a feedback channel back to the GUI.
- GUI side: =sequence_panel::show= returns =Option<String>= when the user
  selects a new value; =App::update= forwards it on the command channel
  and clears the warning label.
- Edge case: combo selection identical to active → no-op, no channel send.

*** NSM save / restore
Per the existing NSM plan section, =st-click= needs Announce + Save/Open.
The state to persist is small:

- Active sequence name.
- (Future) any GUI-set runtime overrides (tempo trim, volume, …) — none today.

Save format: a single =st-click.yaml= (different file from the *config*
YAML — name it =session.yaml= or =state.yaml= to avoid confusion) inside
the NSM-provided session path, containing:

#+begin_src yaml
active_sequence: seven_eight_2_2_3
#+end_src

On Open: read the file, set =active_sequence=, send =SwitchTo= into the
sequencer command channel (same plumbing as the GUI uses). Missing file =
fresh session, fall back to CLI/default resolution.

The =:optional-gui:= capability (per the NSM section's resolved decision)
is independent: NSM controls window visibility via Show/Hide; the sequence
selection lives in Save/Open.

*** Implementation order
1. Refactor =Sequencer= to take a command channel; wire =SwitchTo= end-to-end
   GUI → sequencer (drop the "not implemented" warning in =sequence_panel=).
2. Make CLI =sequence_name= optional with the resolution order above.
3. Add NSM Announce + Save/Open persisting =active_sequence=.
4. Wire =:optional-gui:= Show/Hide once the shared NSM helper from the NSM
   section lands (avoid duplicating the code path).

* GUI [/]
A unified GUI is the next major piece of work. It will sit alongside the
existing CLI-launched JACK clients and provide visual control/feedback for the
whole suite.

** Decisions
- *Toolkit:* =eframe= (egui), following the conventions of the
  =qwertysynth= project. Use eframe's default glow backend unless a reason
  to switch to wgpu appears.
- *Reference codebase:* =./qwertysynth/src/gui/= (a sibling project, not
  part of st-suite — being removed after the GUI plan is captured here).
  Conventions extracted below in [[*Conventions derived from qwertysynth][Conventions derived from qwertysynth]].

** Conventions derived from qwertysynth
All adopted as the house style for st-suite GUI work unless explicitly
overridden below.

*** Crate choice & versions
- =eframe= (not raw egui). *Pin =eframe = "0.34"= per-crate* (matches
  qwertysynth and verified by the st-loop spike). Bump deliberately;
  the trait surface has changed across recent versions (see below).
- Default backend (glow). No wgpu, no =egui_extras=, no =rfd= unless a
  concrete need appears.
- =crossbeam-channel= for all GUI ↔ worker thread messaging.

*** eframe 0.34 API specifics (learned from the st-loop spike)
- =impl eframe::App= must define
  =fn ui(&mut self, ui: &mut egui::Ui, frame: &mut eframe::Frame)=.
  =fn update(&mut self, ctx: &egui::Context, frame: &mut eframe::Frame)=
  exists but is =#[deprecated]= in 0.34 — do *not* use it.
- Inside =ui()=, panels are added with =.show_inside(ui, ...)=, not
  =.show(ctx, ...)=. Use =Panel::top("id")= / =Panel::bottom("id")= /
  =Panel::left("id")= / =Panel::right("id")=; the standalone
  =TopBottomPanel= / =SidePanel= type aliases are deprecated.
- The =run_native= app-builder closure must return
  =Result<Box<dyn App>, Box<dyn Error + Send + Sync>>=. Wrap in =Ok=:
  #+begin_src rust
  Box::new(move |cc| {
      cc.egui_ctx.set_visuals(egui::Visuals::dark());
      Ok(Box::new(MyApp::new(...)))
  })
  #+end_src
- Import =egui= via =use eframe::egui;= (eframe re-exports it).
- Reference implementation: =st-loop/src/bin/gui_spike.rs= (on the
  =dev= branch). Delete or absorb once real GUI implementation begins.

*** Process / window model
- *Single native window per process,* launched via =eframe::run_native=
  on the main thread. Pop-out config UIs are =egui::Window= opened from
  the central area, *not* additional OS windows.
- =ViewportBuilder::default().with_inner_size([W, H]).with_title(...)=,
  dark visuals (=cc.egui_ctx.set_visuals(egui::Visuals::dark())=) set
  once at construction.
- =main()= is synchronous. If async tasks are needed (JACK, evdev, NSM,
  ...), construct a =tokio::runtime::Runtime= manually and =rt.spawn(...)=
  them before calling =eframe::run_native=.

*** File / module organization
- One module per panel / widget / indicator under =src/gui/=, named by
  suffix:
  - =<thing>_panel.rs=    — composite interactive UI, may own a popup
                            config window.
  - =<thing>_widget.rs=   — larger drawable widget (custom paint, takes
                            real estate).
  - =<thing>_indicator.rs= — tiny status dot / lamp.
  - =<thing>_bar.rs=      — full-width strip (info/status).
- =gui/mod.rs= contains *only* glue: submodule decls, the =App= struct,
  the =eframe::App= impl, the =run()= launcher, and per-frame
  drain/dispatch helpers. *No business logic in =mod.rs=.*

*** Panel / widget implementation pattern
- *No trait, no struct-with-show.* Each module exposes free functions:
  - Read-only:   =pub fn show(ui: &mut Ui, state: &AppState)=
  - Interactive: =pub fn show(ui: &mut Ui, state: &mut AppState) -> R=
    where =R= is whatever the caller needs (=bool= "needs rebuild",
    =Option<NewValue>= for the new param to forward on a channel, ...).
- Panels with a popup config window use the *two-function pattern*:
  #+begin_src rust
  pub fn show_top_bar(ui: &mut Ui, state: &mut AppState) -> bool;
  pub fn show_config_window(ctx: &Context, state: &mut AppState) -> bool;
  #+end_src
  The config window takes =&Context= (not =&mut Ui=) because
  =egui::Window= opens against the context.
- *All widget state lives in =AppState=* — including transient flags like
  =show_<x>_config: bool=. Widgets are pure renderers; nothing persists
  inside them. (Rationale: everything inspectable, trivially serializable
  later if we ever need to round-trip UI state via NSM.)
- Give each popup =egui::Window= a stable =Id= derived from the active
  variant (=Id::new("<panel>_window_<variant>")=) so egui doesn't lose
  layout when the user switches modes.

*** AppState pattern
- A single =struct AppState= holds *all* app state.
- =pub type SharedState = Arc<Mutex<AppState>>;=.
- *The audio / JACK thread never locks =AppState=.* It owns its own
  data and communicates exclusively via channels. The mutex exists for
  GUI-thread + helper-task contention only.
- For "mode" data with per-mode parameters, use an *enum-of-param-structs*:
  one enum variant per algorithm, each carrying its own typed param
  struct. Provide =name() -> &'static str= on the enum for labels.
  Match the variant in both the renderer and the worker.

*** GUI ↔ worker communication
- *=crossbeam_channel::unbounded= exclusively.* No =tokio= channels in
  the GUI <-> audio path.
- *Push, drained by =try_recv= once per frame.* At the top of
  =eframe::App::update=, call a =drain_<x>_events= helper that
  =try_recv=s every relevant channel into local state under a short
  lock, then releases the lock before drawing.
- *Decode/translate in the worker, send high-level state to the GUI* —
  never raw MIDI bytes or raw transport frames. e.g. the audio thread
  decodes sustain CC 64 into a =bool= before pushing.
- The audio / RT thread does only lock-free crossbeam sends; heavy work
  (e.g. recomputing a wavetable, rebuilding a scene) happens on the
  *GUI thread* and the result is pushed back as a finished value the
  audio thread atomically swaps in.
- =ctx.request_repaint()= at the end of every =update()= keeps the GUI
  live so it polls channels each frame. Simple but burns CPU; if a panel
  is idle-heavy, switch to =request_repaint_after(Duration)= or convert
  that surface to event-driven repaints.

*** Layout idioms
- Prefer =TopBottomPanel::bottom("...")= for status strips; draw the main
  content directly in the central area. Use =SidePanel= only when there's
  a real reason.
- Custom drawing: =ui.allocate_exact_size(...)= then
  =ui.painter().rect_filled / circle_filled / rect_stroke / text=.
- Pull colors from =ui.visuals()= (=widgets.inactive.bg_fill=,
  =selection.bg_fill=, =text_color=) so widgets respect the theme.
  Hardcoded RGBs allowed only for status indicators (red/green lamps).
- Fixed sizing constants at the top of each widget file, with helper
  fns for derived sizes.
- =ComboBox::from_id_salt("<name>_combo")= for stable IDs.
- Time-valued sliders: =.logarithmic(true).text("...").suffix(" s")=.

*** "Needs work" return convention
Panel/widget functions return =bool= or =Option<Value>= indicating
whether the caller should kick off heavy work. =App::update= aggregates
these flags and triggers the rebuild *once at the end of the frame*,
not inline during widget rendering.

*** House style
- Tab indentation (matches qwertysynth and the rest of st-suite).
- TODO marker style: inline =// TODO: <concrete next step>= comments
  with actionable descriptions, not vague aspirations.

** Architecture for st-suite

*** Scope decision (defer — see open questions)
Two viable shapes for the suite GUI. *Pick one before implementation*:
- *(A) One GUI process per component.* Each of st-conductor / st-click
  / st-loop grows its own =--gui= mode (or sibling binary) that opens
  an eframe window owned by that process. Mirrors qwertysynth (process
  owns both audio + GUI). Natural NSM fit: each component is one NSM
  client with an =:optional-gui:= capability.
- *(B) One unified GUI process.* A new =st-gui= submodule with one
  eframe window that talks to all three running components over some
  control bus. The components stay headless. Better single-pane-of-glass
  UX, but adds a control protocol design problem (see next).

Strong lean: *(A)* for the first cut. It matches qwertysynth, avoids
inventing a control bus, keeps audio + GUI in the same process (so the
GUI can use Arc<Mutex<AppState>> + crossbeam exactly as qwertysynth
does), and gives us three small wins instead of one big one. (B) can
be layered on top later as an additional aggregator window.

*** Per-component GUI plans (assuming scope A)

**** st-conductor GUI
- Transport panel: tempo (BPM), time signature (numerator/denominator),
  current BBT readout, transport state (Rolling / Stopped / Starting).
- Beat indicator: a flashing dot driven by the same =next_beat_frame=
  the conductor pushes to st-sync.
- Optional: a sparkline of frames-per-beat over time for debugging.
- Channels: the JACK timebase callback already updates an
  =Arc<AtomicU64>=; add an =Arc<AtomicXxx>= or crossbeam channel for
  current BBT / state and have the GUI poll once per frame.

**** st-click GUI
- Sequence selector: combo box listing top-level keys from the loaded
  YAML.
- Beat indicator: same flashing-dot pattern as the conductor, driven
  from =st_sync::client= ticks.
- Tempo / meter readout: read-only, mirrored from JACK transport via
  =st_lib::jack_transport::query_transport= once per frame.
- YAML editor (later): an =egui::TextEdit::multiline= bound to the
  current config file with a "Reload" button.

**** st-loop GUI
- 8×8 scene/track grid (the core view). Each cell shows record/play
  state and (later) a level meter.
- Track strip: input gain, output level, mute / solo.
- Scene controls: play / stop, undo, clear, go — duplicating the MIDI
  PC vocabulary so a touch screen / mouse can drive the looper.
- Transport readout: bar / beat from JACK transport.
- NSM-aware: respect the path NSM provides; "dirty" flag flips on
  state changes.

*** Control / sync paths (scope A specifics)
- *No new control protocol.* Each component's GUI talks to its own
  audio thread via crossbeam, exactly as qwertysynth does.
- *Cross-component awareness:* GUIs can read JACK transport (tempo /
  BBT) via =st_lib::jack_transport= and the next-beat tick via
  =st_sync::client::Client=. That covers most of what each GUI needs
  to mirror the others. No bespoke RPC needed for v1.

* NSM (Non Session Manager) [/]
Each st-suite component should be individually NSM-compatible so the suite
can be started/saved/restored as a single session by an NSM host
(=nsm-legacy-gui=, =agordejo=, =RaySession=, ...).

** Audit current NSM compatibility
- [X] *st-loop* — already wires NSM Announce/Save/Open (=src/nsm.rs=,
  =dispatcher.rs:process_save_request/process_load_request=). Audit
  findings (nsm branch):
  - [X] *WAV persistence* — =sequence.rs:save= writes 32-bit float
    stereo WAV via =hound= to =<session_path>/<filename>=, where
    =filename= is =<track>-<epoch>.wav=. Load reads via =WavReader=.
    *Round-trip bug:* =sequence.rs:load= sets =self.filename = file=
    where =file= is the full path the dispatcher passed in
    (=format!("{}/{}", path, item.filename)=). After load, =self.filename=
    is the absolute path; a subsequent Save would write
    =<path>/<full_path>=. Fix: either (a) =load= should accept
    =(dir, basename)= and store only the basename, or (b) strip the
    dir back off in =load=.
  - [X] *Save guard* — =sequence.rs:save= refuses to overwrite an
    existing file (silent return). Fine for v1 since recordings are
    immutable, but worth a log warning, not silent.
  - [X] *Off-by-one in scene save* — =dispatcher.rs:process_save_request=
    iterates =0..(self.scenes.len() - 1)=, dropping the last scene. Should
    be =0..self.scenes.len()=.
  - [X] *Announce capabilities* — currently sent as =""=. Should advertise
    =:switch:= (the dispatcher already supports re-Open by replacing
    =self.path= and re-loading) and later =:optional-gui:= once GUI lands.
    =:dirty:= requires implementing =/nsm/client/is_dirty= /
    =/nsm/client/is_clean= broadcasts on state change — defer until
    needed.
  - [X] *Path honored* — yes. Save writes inside =self.path=, Load reads
    from inside =self.path=. CWD not used. ✅
  - [X] *Reply ordering* — =/reply= for Save and Open is sent immediately
    after queuing the work onto the dispatcher channel, *before* the
    work actually completes. NSM spec wants the reply *after*. Most
    hosts tolerate this; revisit when extracting the common client.
  - [X] *Socket model* — =std::net::UdpSocket= used inside a
    =tokio::task::spawn=d async task (blocking =recv_from= holds the
    runtime). Extracted version should use =tokio::net::UdpSocket=.

  tempo, time signature (and any future transport state).
- [ ] *st-click* — no NSM today. Needs Announce + Save/Open persisting the
